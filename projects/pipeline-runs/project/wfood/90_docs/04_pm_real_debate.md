## Topic

Brief 要求构建一个外卖模拟平台（wfood），包含登录注册、店铺浏览、购物车、下单流程，纯前端 Vue3 + Vite，拟物化 UI，预留后端接口。核心评估：四个功能模块叠加的体量是否可控，以及"模拟平台"和"预留后端接口"这两个定义隐含的架构决策。

## Part B：需求复杂度诊断

### 核心问题：四个模块 = 一个完整应用

"登录注册 + 店铺浏览 + 购物车 + 下单流程"不是四个独立功能，而是一个完整电商应用的四个核心流程。它们之间存在严格的数据依赖：

```
登录 → 浏览店铺 → 查看商品 → 加入购物车 → 编辑购物车 → 确认订单 → 提交订单
```

这意味着每个模块不是独立开发后拼接的，而是需要统一的数据流和状态管理。体量评估：**中等偏大项目。**

### 功能逐项评估

| 模块 | 最小版本 | 标准版本 | 膨胀版本 | 风险 |
|---|---|---|---|---|
| 登录注册 | 表单 + 模拟校验 + 路由守卫 | + 表单验证规则 + 记住登录 + 错误处理 | + 手机号验证 + 第三方登录 + 找回密码 | 低，但膨胀方向多 |
| 店铺浏览 | 店铺列表页 + 店铺详情页（商品列表） | + 分类筛选 + 搜索 + 排序 | + 评价系统 + 距离排序 + 营业状态 | **中-高**，页面数量多 |
| 购物车 | 本地购物车（增删改数量 + 价格计算） | + 店铺级购物车隔离 + 清空 | + 优惠券 + 凑单提示 + 跨店购物车 | 中 |
| 下单流程 | 确认订单页 + 提交（模拟） | + 地址选择 + 备注 + 支付方式（模拟） | + 订单跟踪 + 历史订单 + 退款 | **中**，流程页数量多 |

### 核心复杂度风险

#### 1. 页面数量——高风险 ⚠️⚠️

这是最现实的体量风险。一个外卖平台最少需要以下页面：

| 页面 | 必要性 |
|---|---|
| 登录页 | 必须 |
| 注册页 | 必须 |
| 店铺列表页 | 必须 |
| 店铺详情页（商品列表） | 必须 |
| 购物车页 | 必须 |
| 确认订单页 | 必须 |
| 下单成功页 | 建议有 |
| 个人中心（订单历史） | 可选 |

**最少 6-7 个页面，** 每个页面需要路由配置、组件开发、状态对接。这是纯前端项目中偏大的页面数量。

**替代路径：** 明确每个页面的最小交付粒度，砍掉非必须页面（如订单历史、个人中心），第一版控制在 6 个核心页面以内。

#### 2. 模拟数据管理——中等风险 ⚠️

"模拟平台"意味着没有真实后端，所有数据需要前端模拟。但"预留后端接口"意味着架构上要设计为可替换的。

- **方案 A：纯本地 Mock 数据（JSON 文件）。** 最简单，数据写死在代码中。但无法模拟真实的增删改操作（如加入购物车后刷新页面数据丢失）。
- **方案 B：LocalStorage + Mock 数据初始化。** 数据持久化到浏览器本地，增删改操作真实生效。比方案 A 更像"模拟"。
- **方案 C：Mock Service Worker（MSW）或类似拦截层。** 在前端模拟 HTTP 请求/响应，架构上与真实后端接口一致。实现成本最高，但"预留接口"的价值最大。

**替代路径：** 建议方案 B。LocalStorage 存储购物车和订单数据，Mock JSON 存储店铺和商品数据。既保证操作真实感，又不需要引入额外依赖。"预留接口"通过在代码中定义 API 调用层（函数名和参数约定与真实 API 一致）来实现，后续替换后端时只需修改调用层实现。

#### 3. 拟物化 UI 工作量——中等风险 ⚠️

- 拟物化（Skeuomorphic）设计比扁平化设计需要更多 CSS 工作：阴影层次、纹理模拟、立体按钮、渐变效果。
- 每个组件（按钮、输入框、卡片、导航栏）都需要拟物化风格的样式。
- Vue3 组件化可以复用，但初始的组件样式库搭建成本不低。

**替代路径：**
- 定义一套核心拟物化 Token（阴影值、圆角值、渐变色、边框样式），所有组件共享。避免每个组件独立设计。
- 如果时间紧张，可以只在关键组件（按钮、卡片、导航栏）上做拟物化，列表项等次要组件用简化样式。

#### 4. 状态管理复杂度——中等风险 ⚠️

四个模块之间有数据流依赖：
- 登录状态 → 路由守卫 + 全局用户信息
- 购物车 → 店铺级隔离（不同店铺的商品在不同购物车）→ 全局状态
- 下单流程 → 从购物车读取 → 生成订单 → 清空购物车 → 写入订单历史

需要一个全局状态管理方案。Vue3 的选择：
- **Pinia（推荐）**：Vue3 官方推荐，轻量，TypeScript 友好。
- **Vuex**：Vue2 时代的方案，Vue3 仍支持但已不是首选。
- **Provide/Inject + Reactive**：最轻量，但跨组件状态共享不优雅。

**替代路径：** 用 Pinia。建三个 Store（UserStore、CartStore、OrderStore），各自管理一个领域的状态。

## Position A

**立场：** Brief 的四个模块功能辨识度足够，但每个模块的粒度必须由 pm-core 在收口时明确，否则工程师会自行决定"做到什么程度"，导致交付偏差。

**主要理由：**
- "店铺浏览"可以是一个列表页，也可以是列表 + 详情 + 筛选 + 搜索 + 排序 + 评价的完整子系统。粒度差距 5 倍以上。
- "下单流程"可以是一个确认页，也可以是确认 + 地址 + 备注 + 支付选择 + 结果页的多步流程。
- Non-Goals 的缺失意味着没有膨胀刹车。外卖平台的自然膨胀方向极多（评价、优惠券、地址管理、订单历史、搜索…），没有 Non-Goals 工程师不知道在哪里停下来。

**若采纳，需要保住什么：**
- 四个核心流程的完整闭环（登录 → 浏览 → 购物车 → 下单）不能断裂。
- "预留后端接口"的架构策略。

---

## Position B

**立场：** 如果确认是"模拟平台"而非"真实产品"，可以接受更激进的范围收缩，将第一版定义为"能演示完整购买流程的最小可用版本"。

**主要理由：**
- "模拟平台"的用户预期是"体验流程"，不是"真正下单吃饭"。这意味着很多真实产品的必需功能（地址管理、支付、订单跟踪）在模拟场景下可以大幅简化。
- 模拟数据意味着不需要处理真实业务的异常情况（库存不足、支付失败、配送超时）。

**若采纳，需要放弃什么：**
- 完整电商体验的预期。
- 搜索/筛选/排序功能。
- 订单历史页面。
- 地址管理。

**若采纳，需要保住什么：**
- 完整的购买流程闭环（登录 → 找到商品 → 加入购物车 → 下单）。
- 拟物化 UI 风格（Brief 明确要求）。
- Vue3 + Vite 技术栈。

## What Seems Most Valid

1. **完整购买流程闭环是核心价值。** 用户能走通"登录 → 浏览 → 选商品 → 加购物车 → 下单"的全流程，这就是模拟平台的意义。
2. **拟物化 UI 是差异化要求。** Brief 明确指定，应该保留，但需要控制在关键组件上。
3. **"预留后端接口"是好的架构策略。** 通过 API 调用层抽象，后续切换真实后端成本低。
4. **店铺浏览是最容易膨胀的模块。** 需要明确粒度。

   - 搜索/筛选/排序

   - 订单历史/个人中心
