## Topic

外卖模拟平台 wfood——登录注册、店铺浏览、购物车、下单流程，Vue3+Vite，拟物化 UI，绿色主色调 + 橙色重点色。核心问题是：**四条功能线（登录注册、店铺浏览、购物车、下单）是外卖产品的标准骨架，但拟物化 UI 这个风格选择把实现复杂度和体验风险同时拉高了——拟物化做得好是辨识度，做得差是廉价感。如何确保它成为产品记忆点而不是产品负担？**

## Position A：拟物化 UI 是产品的最大体验赌注

### 立场

Brief 明确选择了"拟物化 UI"——这是一个高辨识度但高风险的风格方向。2026 年的主流 Web 产品几乎全部采用扁平化或 Material 风格，拟物化是一种反潮流选择。**如果做得好，产品立刻有记忆点；如果做得不到位，会让人觉得"这是 2012 年的界面"。**

### 主要理由

1. **拟物化的"到位"和"不到位"差距是悬崖式的**——好的拟物化：按钮有轻微的立体凸起感、卡片有自然的投影层次、输入框有内凹的凹槽质感、图标有微妙的光泽变化。差的拟物化：到处是笨重的渐变、过度的阴影、粗糙的纹理，看起来像早期 iOS 的廉价模仿。**中间状态几乎不存在。**

2. **拟物化在组件层面的实现成本远高于扁平化**——一个扁平化按钮只需要 `background-color + border-radius`。一个拟物化按钮需要：多层渐变（高光 + 底色 + 暗部）、内阴影（凹陷感）、外阴影（悬浮感）、hover 态的渐变方向变化、active 态的按下效果。每个交互组件的 CSS 工作量可能是扁平化的 3-5 倍。

3. **绿色 + 橙色的配色组合在拟物化语境下需要特别处理**——在扁平化中，`#4CAF50` 和 `#FF9800` 直接用就很好看。但在拟物化中，每个颜色都需要对应的高光色、暗部色、渐变过渡色。绿色按钮的高光可能是 `#66BB6A`，暗部可能是 `#388E3C`——这需要一套完整的拟物化色彩系统，不只是两个色值。

4. **拟物化风格在不同组件间的一致性是巨大挑战**——登录页的输入框、店铺列表的卡片、购物车的商品条目、下单页的表单——所有这些组件都需要统一的拟物化语言。如果登录页做得精致但购物车页面突然变扁平，产品就碎了。

### 可能收益

如果拟物化执行到位，wfood 在视觉上会和所有外卖产品形成强烈区分——用户会记住"那个看起来像实物的外卖 App"。

### 可能代价

实现成本高、一致性维护难、风格过时风险（如果市场不接受拟物化审美）。

### 若采纳，需要保住什么

- 拟物化的视觉语言必须在全产品中一致——不能有的页面拟物有的扁平
- 核心交互组件（按钮、输入框、卡片、开关）的拟物化质感必须到位
- 配色系统需要完整的拟物化色彩梯度，不只是两个主色

## Position B：四条功能线是外卖产品的"标配"，风险在于做得和竞品一样

### 立场

登录注册 + 店铺浏览 + 购物车 + 下单流程——这是任何一个外卖产品的标准骨架。美团、饿了么、DoorDash 都有这四条线。**如果 wfood 只是把这四条线用拟物化风格重新画一遍，它就只是一个"好看的外卖壳子"，没有产品灵魂。**

### 主要理由

1. **"模拟平台"暗示不需要真实后端，但也暗示不需要真实交易**——这意味着 wfood 的价值不在于"真的能点外卖"，而在于"体验一个外卖平台的完整流程"。那么问题来了：**体验的目的是什么？** 是展示前端能力？是做一个作品集项目？是教学用途？不同目的决定了不同的体验侧重。

2. **店铺浏览是最容易做出差异化的环节**——登录注册和下单流程在所有产品中都高度相似，但"怎么逛店"可以有完全不同的体验。列表式？卡片式？地图式？分类筛选？推荐排序？这个环节是用户停留时间最长的地方，也是最能体现产品感的地方。

3. **购物车的交互细节是"顺手"和"别扭"的分水岭**——添加商品到购物车的动画、购物车角标的数字变化、滑动删除商品、修改数量的 stepper 控件、购物车展开/收起的交互——这些细节做对了用户不会注意，做错了用户会立刻感到不适。

4. **缺少搜索功能是一个明显的体验缺口**——外卖平台的核心使用路径之一是"搜店铺/搜菜品"。如果只有浏览没有搜索，用户面对大量店铺时会感到无力。但 Brief 没有提到搜索。

### 可能收益

在某一条功能线上做出超越"标配"的体验，产品就有记忆点。

### 可能代价

增加功能深度会扩大 Scope。

### 若采纳，需要放弃什么

- 四条线平均发力的幻想——建议选择"店铺浏览"作为体验重点，其他三条保持基本完成度

## Position C：Brief 缺少关键定义，pm-core 收口前需要补充

### 立场

当前 Brief 的信息量不足以让 pm-core 做出完整的产品定义。以下缺口需要补齐：

| 缺失字段 | 影响 |
|---|---|
| 目标用户 | 决定交互复杂度和信息密度 |
| 模拟数据规模 | 几家店铺 vs 几百家？决定列表/筛选方案 |
| Non-Goals | 是否需要搜索？评价？订单历史？配送追踪？ |
| 拟物化程度 | 轻度拟物（按钮有质感）vs 全面拟物（所有组件立体化）？ |
| 响应式/移动端 | 手机端需要适配吗？外卖场景以移动端为主 |
| Success Criteria | 做到什么程度算完成？ |

### 主要理由

1. **"预留后端接口"意味着数据层设计会影响前端架构**——如果只是硬编码 mock 数据，和设计一套可切换的数据适配层（当前用 mock，未来接真实 API），代码结构完全不同。这个决定应该在产品定义阶段明确。

2. **移动端适配是外卖产品的默认需求**——外卖是典型的移动端场景。如果 wfood 只做桌面端，需要明确说明为什么。如果要做移动端，拟物化 UI 在小屏幕上的表现需要特别验证。

3. **模拟数据需要有真实感**——"外卖模拟"如果店铺只有 3 家、每家只有 5 个菜品，浏览体验会很空。需要有足够的模拟数据让店铺浏览环节有内容密度。

## What Seems Most Valid

1. **功能骨架覆盖了外卖产品的完整用户流程**——从进入（登录）到浏览到选择到下单，链路完整。
2. **Vue3 + Vite 技术栈合理**——现代前端框架，适合组件化开发，拟物化组件可以封装复用。
3. **绿色 + 橙色配色在外卖场景有天然适配性**——绿色暗示新鲜/健康/外卖品牌色（参考美团），橙色暗示食欲/活力。方向正确。
4. **"预留后端接口"说明有长远考虑**——不是做一个一次性 demo，而是为后续扩展做准备。

---

## What Still Needs Evidence

1. **拟物化的具体程度**——是所有组件全面拟物化，还是只在关键交互元素上做拟物化点缀？
2. **模拟数据的规模和内容**——多少家店铺？每家多少菜品？有没有分类？
3. **移动端适配需求**——外卖是移动端优先场景，是否需要适配？
4. **目标用户和使用场景**——展示项目？教学？实际使用？
5. **店铺浏览的具体形式**——列表？卡片？有无筛选/分类？
6. **是否需要搜索功能**——外卖产品的核心功能之一
7. **订单流程的深度**——选择地址？选择支付方式？订单确认页？还是简化的下单？

## Recommendation to pm-core

### 建议收口方向

1. **收口拟物化程度为"轻度拟物"**——核心交互组件（按钮、输入框、卡片、底部导航）做拟物化质感，但不要让整个页面都充满立体效果。建议参考 iOS 6 时代的设计语言，但更克制：有阴影层次但不过度，有质感但不加纹理。理由：
   - 全面拟物化的 CSS 工作量可能占整个项目的 40% 以上
   - 轻度拟物既能有辨识度，又能控制成本
   - 一致性更容易维护

2. **把"店铺浏览"作为体验重点**——这是用户停留时间最长、最能体现产品感的环节。建议：
   - 店铺卡片有拟物化的立体感（轻微凸起 + 自然阴影）
   - 菜品图片展示区有质感边框
   - 分类筛选做成交互流畅的 tab 或抽屉
   - 店铺详情页的菜品列表有良好的信息层级
   - 登录注册和下单流程保持基本完成度即可，不需过度投入

3. **购物车交互必须顺手**——核心细节：
   - 添加商品到购物车时有"飞入"动画（商品图标飞向购物车角标）
   - 购物车角标数字有弹跳变化
   - 购物车展开/收起有流畅过渡
   - 数量修改用 +/- stepper，不要用下拉框
   - 滑动删除或左滑删除商品

4. **收口模拟数据规模**——建议至少 8-12 家店铺，每家 8-15 个菜品，有分类（主食、小吃、饮品等）。数据太少会让浏览体验空洞，太多会让 mock 数据维护成本过高。

5. **移动端必须适配**——外卖是移动端优先场景。建议采用 mobile-first 响应式设计，桌面端做宽度适配即可。如果 Brief 的"纯前端"意味着需要一个可以部署到手机上浏览的页面，移动端体验就不能退让。

6. **建议加入搜索功能（可退让）**——即使只是前端过滤式的简单搜索（按店铺名/菜品名），也比完全没有搜索更符合用户对外卖平台的预期。成本低，体验提升明显。

7. **"预留后端接口"需要收口为具体的接口契约**——建议 pm-core 在 Product Spec 中定义 mock 数据的结构和前端的数据调用方式（composable / service 层），确保后续接真实后端时改动最小。

### 建议暂缓 / 可退让

- **全面拟物化**——退让为轻度拟物，聚焦核心组件
- **搜索功能**——有则更好，可退让
- **订单历史/评价系统**——不在当前 Scope
- **配送追踪模拟**——不在当前 Scope
- **地址管理**——可以简化为一个输入框，不需要完整地址管理
- **支付方式选择**——可以简化为模拟选择，不需要真实支付流程
- **店铺详情页的深度**——基本菜品展示 + 加入购物车即可，不需要评论/评分/营业时间等
- **动画丰富度**——飞入动画有则更好，没有不阻塞
- **暗色模式**——不在当前 Scope

### 强主张

- **拟物化的视觉一致性必须在全产品中保持**——登录页、首页、购物车、下单页的拟物化风格不能有断裂。如果做不到一致，宁可降低拟物化程度也不能让某些页面"掉线"。
- **购物车的交互反馈必须即时且愉悦**——添加商品 → 角标变化 → 购物车内容更新，这条链路的每一步都需要视觉反馈。没有反馈的购物车是"死"的。
- **移动端适配是外卖产品的默认要求**——如果 wfood 只做桌面端，需要在产品定义中明确说明理由。否则默认需要移动端适配。
- **模拟数据必须有真实感**——店铺有名称、头像、评分、月售、配送费、预计时间；菜品有名称、图片、价格、月售。如果 mock 数据太假，整个模拟感就崩了。

### 不建议退让项

- 拟物化风格的全产品一致性
- 购物车的交互反馈完整性
- 移动端适配
- 模拟数据的丰富度和真实感
- 店铺浏览环节的体验深度

### 可退让项

- 拟物化程度（轻度拟物 > 全面拟物）
- 搜索功能
- 动画丰富度
- 订单流程的深度（简化地址和支付）
- 店铺详情页的信息丰富度
- 登册注册的体验精致度
