## Topic

Brief 要求构建"金铲铲之战经济模拟器"，包含三个交付物：详细数学建模、连胜连跪策略最优解探讨、论文级文档 + 交互可视化网站。核心评估：这是一个**研究 + 工程双轨并行**的项目，数学建模的深度和可视化网站的复杂度都需要约束，否则两头都做不好。

## Part A：Brief 质量诊断

### 信息密度

本 Brief 虽然只有一行 Goal，但信息密度较高——指明了研究对象（金铲铲之战经济系统）、研究方向（连胜连跪策略最优解）、交付形式（论文级文档 + 交互可视化网站）。方向可辨识。

## Part B：需求复杂度诊断

### 关键认知：本项目有两个交付物，复杂度来源完全不同

| 交付物 | 性质 | 复杂度来源 | 风险类型 |
|---|---|---|---|
| 论文级文档 | 研究型 | 数学建模的深度、最优解的证明/验证 | **范围不可控**——研究深度没有天然上限 |
| 交互可视化网站 | 工程型 | 参数调节、实时计算、图表渲染 | **工程量可控**——但依赖研究成果的输出 |

两者的关系：**文档是主体，网站是文档的交互式呈现。** 如果没有清晰的数学模型，网站就没有可展示的内容。因此，**必须先收口研究范围，再定义可视化范围。**

---

### 核心复杂度风险

#### 1. 研究范围不可控——最高风险 ⚠️⚠️⚠️

金铲铲之战的经济系统是一个多变量、多约束的优化问题。涉及的变量可能包括：

| 变量 | 说明 | 建模难度 |
|---|---|---|
| 基础金币收入 | 每回合固定收入 | 低 |
| 利息机制 | 每 10 金币 1 利息，上限 5 | 低 |
| 连胜奖励 | 3/5/8 连胜的不同奖励 | 低-中 |
| 连败奖励 | 3/5/8 连败的不同奖励 | 低-中 |
| 经济上限 | 金币存储上限 | 低 |
| D 牌成本 | 每次刷新 2 金币 | 低 |
| 升级成本 | 不同等级升级所需金币 | 低 |
| D 牌概率 | 不同等级的各费用卡概率分布 | **中-高** |
| 阵容构建 | 买什么卡、什么时候升人口 | **高** |
| 血量/排名 | 连败策略的血量成本 | **中-高** |
| 对手行为 | 其他玩家的策略影响 | **极高** |

**如果不对变量范围做硬性约束，建模可以无限深入。** "连胜连跪策略最优解"听起来聚焦，但"最优解"的定义需要明确：在什么约束下？考虑哪些变量？纯理论分析还是蒙特卡洛模拟？

**替代路径：**
- **硬性约束建模范围。** 建议第一版只建模"纯经济变量"（金币收入、利息、连胜/连败奖励），不涉及 D 牌概率、阵容构建、对手行为。
- 将"最优解"降级为"策略对比分析"——对比几种典型策略（保守攒利息 vs 激进连胜 vs 连败转型）的经济收益，用蒙特卡洛模拟验证，不做严格的数学证明。
- 如果需要严格最优解，这本身就是一个研究课题，不适合在工程 pipeline 中并行推进。

#### 2. "论文级文档"的深度歧义——高风险 ⚠️

"论文级"可以指：
- **方案 A：结构严谨、论证完整的技术文档。** 含问题定义、假设、模型、推导、结果、结论。不需要同行评审级别的严谨性。
- **方案 B：真正可投稿的学术论文。** 含文献综述、方法论、严格证明、实验设计、统计检验。

两者的投入差距是 **10 倍以上。** Brief 没有定义。

**替代路径：** 建议定义为方案 A——"结构严谨的技术分析文档"，包含问题定义、数学模型、策略对比、模拟结果、结论。用 Markdown 或 LaTeX 排版，视觉上接近论文格式，但不需要学术论文的全部要素（文献综述、严格证明、统计检验）。

#### 3. 可视化网站的范围依赖研究输出——中等风险 ⚠️

- 网站展示什么取决于数学模型的输出。如果模型还没定，网站无法开始开发。
- 网站的交互深度依赖参数数量。如果模型有 10+ 个可调参数，交互界面的复杂度会显著上升。

**替代路径：**
- 先完成数学建模（文档），再根据模型输出定义可视化内容。
- 网站的交互参数限制在 3-5 个核心变量（如：初始金币、连胜/连败策略选择、利息阈值），避免参数爆炸。

#### 4. 最优解的可解性存疑——中等风险 ⚠️

- TFT 经济系统是一个多阶段决策问题，且存在对手交互（博弈论层面）。严格的最优解在考虑对手行为时可能是 NP-hard 或不可解的。
- 即使不考虑对手行为，纯经济变量的最优策略也可能没有闭式解，只能通过数值方法近似。

**替代路径：**
- 放弃"严格最优解"的定位，转为"策略对比 + 模拟验证"。用蒙特卡洛模拟对比几种典型策略，给出"在什么条件下哪种策略更好"的结论。这对目标受众（TFT 玩家/数学爱好者）比严格的数学证明更有价值。

## Position A

**立场：** 研究范围必须硬性收口，否则"数学建模"会无限膨胀。建议将建模范围限制在纯经济变量，"最优解"降级为策略对比分析。

**主要理由：**
- TFT 经济系统的完整建模（含 D 牌概率、阵容构建、对手行为）是学术级研究课题，不适合在工程 pipeline 中并行推进。
- "连胜连跪策略"的分析如果只考虑纯经济变量（金币收入、利息、连胜/连败奖励），可以用相对简单的模型完成，结果对目标受众仍然有价值。
- 蒙特卡洛模拟 + 策略对比的结论，比试图求"严格最优解"更实际、更可交付。

**若采纳，需要保住什么：**
- 数学模型的结构完整性（问题定义 → 假设 → 模型 → 模拟 → 结论）
- 策略对比的可视化展示
- 文档的专业排版和可读性

---

## Position B

**立场：** 如果目标是真正的学术输出，应该将"论文"和"可视化网站"拆分为两个独立项目，论文优先、网站后置。

**主要理由：**
- 论文的深度和网站的工程量是两种不同性质的工作，同步推进容易互相拖累。
- 如果论文需要严格的数学证明或蒙特卡洛模拟，研究成果的输出时间不可预测，网站的开发会因等待研究成果而阻塞。
- 拆分后，论文可以走研究 pipeline（research 主导），网站走工程 pipeline（architect + engineer 主导），各自有明确的交付标准。

**若采纳，需要放弃什么：**
- "同步交付论文 + 网站"的预期。

**若采纳，需要保住什么：**
- 最终仍然有论文 + 网站两个交付物，只是分阶段完成。

4. **论文和网站可以共享数学模型。** 模型只建一次，文档用文字/公式/图表呈现，网站用交互组件呈现。

## Why This Is More Stable

1. **建模范围硬性限定在纯经济变量。** 变量少、约束明确、模型可解、结论可交付。不会陷入"越建越深出不来"的困境。
2. **"策略对比"替代"最优解"。** 蒙特卡洛模拟 + 3-4 种策略对比，可以在合理时间内给出有结论的输出。不需要证明全局最优。
3. **文档和网站共享同一模型。** 模型只建一次，文档用文字/图表呈现，网站用交互组件呈现。避免两头重复建模。
4. **先文档后网站的顺序。** 模型和结论确定后，网站只是"展示形式"，开发节奏可控。
5. **交互参数限制在 3-5 个。** 避免参数爆炸导致的界面复杂度和计算性能问题。

## Recommendation to pm-core

### 必须收口的问题（按优先级）

1. **建模范围：确认只包含纯经济变量（基础金币 + 利息 + 连胜/连败奖励）。** 排除 D 牌概率、阵容构建、对手行为。这是最重要的决策，直接影响项目是否可控。

2. **研究方法：确认"策略对比 + 蒙特卡洛模拟"替代"严格最优解"。** 定义 3-4 种典型策略，用模拟对比经济收益。

3. **文档定义：确认为"结构严谨的技术文档"，不是学术论文。** 包含：问题定义、假设、模型、模拟、结论。不含：文献综述、严格证明、统计检验。

4. **开发顺序：确认先完成文档（建模 + 模拟），再开工网站。** 网站的交互内容依赖模型输出。

5. **网站交互参数：确认 3-5 个核心参数。** 建议：回合数、初始金币、连胜/连败策略选择、利息策略阈值。

### 不需要做的

- 不需要 research 并行（TFT 经济机制是公开信息，不需要外部调研）
- 不需要 pm-ideal 并行（可视化方案没有体验分歧）
- 不需要多轮产品确认
