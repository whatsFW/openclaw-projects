# 决策记录：TFT 经济模拟器

## D1：双交付物 — 论文文档 + 独立网站

- **决定：** 交付两份产物——① 结构严谨的技术分析文档 ② 独立的交互可视化网站
- **原因：** 网站独立于论文，用户覆盖面更广，体验设计不受论文结构束缚
- **影响：** 开发顺序先文档（建模+模拟）后网站（可视化+交互）
- **状态：** ✅ 已落地

---

## D2：研究方法 — 策略对比 + 蒙特卡洛模拟

- **决定：** 采用策略对比 + 蒙特卡洛模拟，而非严格最优解证明
- **原因：** 多阶段博弈问题严格最优解不可达，策略对比 + 模拟对目标受众（TFT 玩家）更有价值
- **影响：** 模型输出为策略间的对比数据，而非"最优解"
- **状态：** ✅ 已落地

---

## D3：建模范围 — 纯经济变量

- **决定：** 硬性限定只建模经济变量（基础金币 + 利息 + 连胜/连败奖励），排除 D 牌概率、阵容构建、对手行为
- **原因：** 保证模型可分析、可解释，避免复杂度过高导致不可控
- **影响：** 模型结论仅在纯经济维度有效，不涉及阵容强度
- **状态：** ✅ 已落地

---

## D4：文档定位 — 技术分析文档，非学术论文

- **决定：** 输出为结构严谨的技术分析文档，不追求学术论文格式
- **原因：** 目标受众是游戏玩家而非学术审稿人，技术分析文档更实用
- **影响：** 文档侧重可读性和实用性，不追求严格的引用和证明
- **状态：** ✅ 已落地

---

## D5：技术架构 — model/ + web/ 分离

- **决定：** `model/` 负责建模引擎和模拟（纯 JS），`web/` 负责可视化和交互（HTML + Charts）
- **原因：** 建模逻辑与展示层分离，便于独立测试和复用
- **影响：** model/ 可独立运行（`node model/test.js`），web/ 依赖 model/ 输出
- **状态：** ✅ 已落地
